# GPU upgrade — AMD Radeon AI PRO R9700 32 GB vs RTX 3060 (local AI)

**Status: OPEN / researching** — not yet decided. Pending (a) second opinions
from other AIs (prompt below), (b) a de-risking proof-of-concept before any buy.
**Date:** 2026-06-01. **Owner:** Jo. **Captured by:** Claude (this session).

## The decision
Whether to replace the **RTX 3060 12 GB (CUDA)** with the **Sapphire AMD Radeon
AI PRO R9700 32 GB GDDR6 (RDNA4, gfx1201)** as the box's single inference GPU —
to unlock a local 27–32B "medium" tier + local RAG over the email/invoice corpus,
and to **fine-tune small models to our narrow, repeatable ops** rather than rely
on general models that hit edge cases.

## Current state (verified 2026-06-01)
- RTX 3060 12 GB, driver 595.71.05. `ollama/ollama:0.22.1` (Docker, NVIDIA CDI
  passthrough `nvidia.com/gpu=all`, `OLLAMA_KEEP_ALIVE=-1`, `NUM_PARALLEL=2`).
- Models: `qwen2.5:7b` (4.7 GB), `phi4:14b` (9.1 GB), `nomic-embed-text`.
- 12 GB = one big model at a time (phi4 alone = 9 GB) → reload thrash.
- ai_usage shows local qwen doing ~the email-classify + invoice-ladder local tier;
  complex work escalates to Claude Haiku/Sonnet.

## R9700 analysis — the core trade
**The prize is 32 GB VRAM (2.7×) + ~640 GB/s (1.8× bandwidth). The risk is ROCm /
RDNA4 (gfx1201) software maturity on Linux — not the hardware.**

| | RTX 3060 (now) | R9700 (proposed) |
|---|---|---|
| VRAM | 12 GB | 32 GB |
| Bandwidth | ~360 GB/s | ~640 GB/s GDDR6 256-bit |
| Compute | CUDA (mature) | RDNA4 / ROCm gfx1201 (newer) |
| Power | ~170 W | ~300 W |

### What it unlocks
1. Local **medium tier** (qwen2.5:32b q4 ~20 GB) → most Haiku escalation in-house
   (privacy + headroom, not just cost — current cloud spend is already ~£0.69/30d).
2. No model thrash — hot + medium + embeddings all resident at once.
3. **Local RAG** over the ~75k backfilled emails (bodies done) + invoices.
4. Big local batch jobs (classify/summarise 5y archive) at zero cloud cost.

### Model recommendation (stay on qwen2.5 family — JSON/tool-use strength)
| Tier | Model | ~VRAM q4 | Job |
|---|---|---|---|
| Hot | qwen2.5:7b (or 14b) | 4.7 / 9 GB | classify, route, fast summaries |
| Medium (new local) | **qwen2.5:32b-instruct q4_K_M** | ~20 GB | invoice extraction, complex JSON, private-doc, reconciliation |
| Embeddings | nomic-embed-text | 0.3 GB | RAG |
| Cloud (kept) | Sonnet/Opus | — | highest-stakes reasoning only |

- **Not 70B** — needs ~40 GB at usable quant; won't fit. 27–32B is the sweet spot.
- Expected: ~15–25 tok/s on 32B, ~40–70 tok/s on 7B (bandwidth-bound). Fine for
  batch/async; acceptable not instant for live chat.

### How it slots into the stack
1. ROCm 6.4+ (`amdgpu-dkms`); confirm gfx1201 enumerated (maybe
   `HSA_OVERRIDE_GFX_VERSION` until native).
2. Compose: `ollama/ollama:rocm` + `/dev/kfd` + `/dev/dri` + `group_add:[video,render]`
   (replaces the NVIDIA CDI device). Retire `u226-gpu-recover.sh`; write an AMD equiv.
3. Re-tier the router (SPEC §7.3 / `static_context` thresholds) → escalate to
   local-32b before cloud. Update CLAUDE.md/AGENTS.md model table.
4. Upstream unchanged — Open WebUI / litellm / llm-router / n8n all hit
   `http://homeai-ollama:11434`. Only the GPU layer changes.

## Jo's thesis (2026-06-01)
Unsloth + optimisation should let us **fine-tune on the AMD card and overcome the

## Verdict so far
- ✅ Capability/VRAM: clear, large upgrade; unlocks local-medium + RAG + fine-tune.
- ⚠️ Single biggest risk: RDNA4/ROCm + Ollama + (especially) **fine-tuning**
  toolchain maturity on Ubuntu 26.04 / kernel 7.0. The CUDA path "just works" today.
- 💡 Counterfactual: a used **RTX 3090 24 GB** (CUDA, ~£600–700) = less VRAM but
  zero ROCm risk and runs 32B q4 + LoRA fine-tune fine. R9700 wins on VRAM/
  bandwidth/power/warranty; 3090 wins on software certainty.

## Second opinion #1 (another AI) — summary + Claude's critique
**Its take (strong):** reframes the choice to **R9700 (32GB) vs used 3090 (24GB)**;
4090 is the wrong buy. Decisive variable = "do you need a local *dense 32B*?"
Key technical point I'd under-specified: **bitsandbytes 4-bit (QLoRA) is
NVIDIA-only — auto-disabled on AMD**, forcing 16-bit LoRA on the R9700. Also: a
fine-tune PoC on the existing CUDA 3060 transfers to a 3090/4090 but NOT to the
R9700's AMD toolchain. Recommends: run a 7B QLoRA PoC on the 3060 first; if a
LoRA doesn't beat schema-constrained prompting, the fine-tune axis disappears and
the only differentiator is local-32B → R9700 (cheaper). Suggests Vulkan
llama.cpp on AMD to dodge the HIP idle-power bug + FP8 mess.

**Claude's critique:** Agree with the reframe + decision rule. Adds: (1) 32GB
partly *cancels* the no-bitsandbytes hit — a 7B bf16 LoRA fits without 4-bit
quant. (2) The PoC's *strategic finding* (does fine-tuning help at all?) transfers
even though the toolchain doesn't. (3) The stack *already* uses JSON-schema
constrained decoding (AGENTS.md U38) — so "prompting already good enough" was
more likely than assumed. (4) Vulkan means likely off-Ollama; idle power matters
because the box is 24/7 (KEEP_ALIVE=-1).

## PoC result (2026-06-01) — baseline arm, run on the 3060
Ran the **production prompted qwen2.5:7b** (`extract_local`, same `_SYS` + schema)
vs stored Haiku-quality labels on **10 real invoices**:

| field | prompted-7B correct |
|---|---|
| vendor | 0/10 |
| gross | **0/10** (returned null every time) |
| date | 2/10 |
| all-3 | 0/10 |
| **gate pass** | **0/10** |

Independently corroborated by production: of 624 invoices, the **local tier
gate-passed 0%** (9 attempts) — **Haiku did 595**. So **schema-constrained
prompting on a 7B does NOT solve invoice extraction.** (NOTE: this is the
*baseline* arm only — a *fine-tuned* 7B might well match Haiku on these narrow,
recurring vendor formats; that arm was NOT run — it needs a PyTorch+PEFT+CUDA
training env + dataset reconstruction, since `ocr_text` isn't persisted.)

### ⚠️ CORRECTION (re-ran on "try again") — the 0% was a PROMPT artifact
Diagnostics showed the prompted 7B **does** read the invoices (returns
`is_invoice=true`, date, full line items, `parse=OK`, `done_reason=stop`) — but
the **conservative production prompt makes it NULL the headline scalars**
(`vendor_name`, `net`, `vat`, `gross`), which is why it fails the gate. Not a
capability ceiling — a prompt gap.

**A/B test (same 7B, same schema, sharpened prompt), 6 invoices:**
| | gross correct | vendor filled |
|---|---|---|
| OLD prompt (prod) | 0/6 | 0/6 |
| NEW prompt | **4/6** | **6/6** |

The 2 misses are identical + learnable: on informal/handwritten invoices the 7B
picked **the customer ("Ye Olde Malthouse") as the vendor**. So:
- **Free win available now:** fix the production `extract_local` prompt → many
  invoices handled at local tier, less Haiku escalation. Ship to `ladder.py`.
- **Fine-tune thesis VALIDATED:** the 7B is capable (gets lines/dates, and
  vendor/gross with a better prompt); a LoRA on ~50–100 labelled invoices would
  nail the residual customer-vs-supplier confusion. This is exactly what tuning
  to narrow ops is good at.
- **But it's a 7B LoRA — fits a 24GB 3090 OR the R9700.** Fine-tuning does NOT
  require 32GB, so the GPU axis is unchanged: 32GB still only matters for a local
  **dense 32B**. R9700 wins iff you want that.

### What the PoC actually decides (superseded by the correction above)
- **Invoice extraction**: prompted-7B fails (0%) → but **Haiku already does it for
  pennies** (total AI spend ~£0.69/30d). So fine-tuning a 7B for invoices would
  save ~nothing — **low ROI** (only matters for privacy or huge volume growth).
- **Email classification** (the high-volume local task): prompted-7B already
  works in production (~400/day) → **no fine-tune needed**.
- ∴ **Neither high-ROI fine-tune target actually needs fine-tuning** — one's
  already cheap via cloud, the other already works local. The fine-tuning case is
  **weaker than the thesis assumed**, so **don't optimise the GPU purchase for
  fine-tuning.** The differentiator that remains is **local dense-32B** (privacy +
  less API escalation) → **favours the R9700 (32GB, cheaper card).**

## Updated verdict
Lean **R9700** — *if* a local 32B genuinely matters to you (it's the only card
that holds it + it's cheaper). Buy a used **3090** only if you later find a real,
recurring need for robust local fine-tuning. The PoC didn't surface one.
**Still de-risk first**: borrow/return-window test that `ollama:rocm` runs
qwen2.5:32b on the R9700 (gfx1201) cleanly before committing — that ROCm
inference check, not fine-tuning, is now the gating risk.
