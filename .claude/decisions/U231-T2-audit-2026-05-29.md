# U231 T2 — Vault role-password audit (2026-05-29)

**Status:** AUDIT ONLY. Actual password rotation deferred to per-service migration sessions.

## Why audit-only in this batch

The memory `feedback-service-pg-user-audit` claimed V177 roles were `NOLOGIN`. Re-audit shows that's outdated:

```
rolname           | rolcanlogin | rolsuper
homeai_hr         | t           | f
homeai_pipeline   | t           | f
homeai_readonly   | t           | f
metabase_app      | t           | f
owner_role        | f           | f    ← only this and the two below are NOLOGIN
personal_role     | f           | f
trading_role      | f           | f
paperless         | t           | f
```

The `owner_role`/`personal_role`/`trading_role` triplet is the RLS realm-switching pattern (per memory `project-realm-split`) — those SHOULD stay NOLOGIN because they're `SET ROLE` targets, not connection identities.

Rotating any of the LOGIN role passwords mid-batch would require simultaneous:
1. Vault `kv put` with the new password,
2. Postgres `ALTER ROLE … PASSWORD …`,
3. Vault-Agent template + container restart,
4. n8n credential update (where applicable),
5. Service-level smoke test.

Doing all five steps within one hands-off batch, per role, with the risk of locking a service out, is the wrong shape for autonomous work. The migration plan calls for one-service-per-session anyway.

## Action items (deferred to per-service sessions)

For each migration target, the per-session checklist:
- [ ] Generate strong password: `openssl rand -hex 32`
- [ ] Vault write: `vault kv put secret/postgres-roles/<role> password=…`
- [ ] Postgres set: `ALTER ROLE <role> PASSWORD '…';`
- [ ] Vault-Agent template entry in `/home_ai/services/homeai-vault-agent/agent.hcl`:
  ```hcl
  template {
    destination = "/run/secrets/<role>-password"
    perms       = 0444
    contents    = <<-EOT
    {{ with secret "secret/data/postgres-roles/<role>" }}{{ .Data.data.password }}{{ end }}
    EOT
  }
  ```
- [ ] Recreate vault-agent container.
- [ ] Update consuming service's env to read `…_DBPASS_FILE=/run/secrets/<role>-password`.
- [ ] Verify service smoke test.
